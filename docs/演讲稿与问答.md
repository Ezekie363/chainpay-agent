# ChainQuery Agent — 演讲稿与评审问答

---

## 一、项目介绍

### 一句话概括

**ChainQuery Agent 是一个能自主为链上数据查询付费的 AI Agent**——用户提问，Agent 自己决策、自己付钱、自己拿数据，全程无需人工审批任何支付。

### 背景与动机

传统 AI Agent 调用外部 API 时，付费环节始终需要人工介入：提前充值、绑定信用卡、手动授权。这与 Agent"自主执行任务"的核心理念相悖。

我们想回答一个问题：**能不能让 Agent 像人一样，真正自主地为自己使用的服务付费？**

### 项目演示流程

```
用户提问："帮我查 BTC、ETH 价格和当前 gas"
   ↓
DeepSeek LLM 分析意图，决定调用 3 个查询工具
   ↓
每个工具调用触发 x402 支付流程
   ↓
Cobo MPC 钱包自主签名，无需人工审批
   ↓
USDC 在 Base Sepolia 链上真实转账
   ↓
Agent 拿到数据，返回完整回答
   ↓
终端实时显示每笔支付的 tx hash，可在 Basescan 验证
```

## 二、核心技术栈

### x402 协议

基于 HTTP 标准的机器支付协议。服务端在响应中返回 `402 Payment Required`，客户端自动完成支付后重试。对服务方友好——只需一个中间件，无需改动业务逻辑。

### Cobo Agentic Wallet（CAW）

MPC（多方计算）钱包，私钥分片存储于多个节点，任何单点均无法还原完整私钥。Agent 通过 Pact（授权策略）签名，超出策略范围的操作会被自动拒绝，天然具备风险控制能力。

### EIP-3009 TransferWithAuthorization

x402 在 EVM 链上的支付机制。Agent 签名一个链下授权（不上链），Facilitator 验证后执行 `transferWithAuthorization` 完成链上 USDC 转账。签名与转账分离，安全且高效。

### DeepSeek LLM + Tool Calling

LLM 理解用户自然语言意图，自主决定调用哪些工具、以什么参数调用。支付由工具内部自动触发，LLM 层完全无感知——它只关心"我需要数据"，不关心"我要怎么付钱"。

### 技术架构总览

| 层级 | 组件 | 职责 |
|------|------|------|
| 交互层 | Rich CLI + prompt_toolkit | 实时展示支付流水、tx hash |
| 智能层 | DeepSeek tool calling | 意图理解、工具决策 |
| 支付层 | x402 + CawSigner | 触发支付、MPC 签名 |
| 钱包层 | Cobo Agentic Wallet | 策略管控、链下签名 |
| 结算层 | x402 Facilitator | 验证授权、执行链上转账 |
| 数据层 | FastAPI + CoinGecko + RPC | 返回真实市场与链上数据 |

## 三、评审问答（10 题）

---

### Q1. CAW 在这个项目中的核心作用是什么？能换成普通私钥签名吗？

**A：** CAW 是本项目支付能力的唯一来源，不可替换。

普通私钥签名意味着私钥必须明文存在于运行环境中，任何代码漏洞或环境泄露都会导致资产全损。CAW 的 MPC 架构将私钥分片分布于多个独立节点，任何单节点均无法还原完整私钥，签名必须经过多方协作完成。

更关键的是，CAW 的 **Pact 策略**允许精确控制签名权限：只允许对特定合约（USDC verifyingContract）、特定链（Base Sepolia）的 EIP-712 消息签名。超出策略的请求会被直接拒绝，即使 Agent 代码被攻击也无法越权操作。这是普通私钥方案完全做不到的风险隔离。

---

### Q2. Agent 是如何"自主"决策的？是规则触发还是 LLM 驱动？

**A：** 完全由 LLM 的 tool calling 机制驱动，没有任何硬编码规则。

用户输入自然语言问题，DeepSeek 模型根据语义理解决定：是否需要查询数据、需要查哪些数据、以什么参数查询。我们没有写任何 `if "价格" in question` 这样的规则。

这意味着 Agent 具备真正的泛化能力——用户说"现在适合发交易吗"，Agent 会自主判断需要先查 gas 价格再给出建议，而不是等待我们预定义这个路径。

---

### Q3. x402 协议的支付流程是怎样的？为什么不直接调用合约？

**A：** x402 的核心优势是**将支付嵌入 HTTP 协议层**，对应用层透明。

流程：① 客户端发起请求 → ② 服务端返回 402 + 支付规格 → ③ 客户端签名授权 → ④ 携带授权头重试 → ⑤ 服务端调用 Facilitator 验证并执行链上转账 → ⑥ 返回数据。

直接调用合约的问题是：需要先上链确认才能拿到数据，延迟高（出块时间 2s+），且需要 ETH 支付 gas。x402 使用 EIP-3009 的链下授权机制，Facilitator 负责执行，调用方无需持有 ETH，延迟主要来自 MPC 签名（约 15-20s），链上转账异步完成。

---

### Q4. 15-25 秒的签名延迟是否可接受？生产环境怎么优化？

**A：** 在当前 Demo 场景下可接受，终端有 `thinking...` 提示，用户体验有预期管理。

延迟来源主要是 CAW 的 MPC 协调：多个节点需要通信并协作完成签名，网络延迟是主要瓶颈。

生产环境的优化方向：
1. **预签名池**：对常用金额提前生成一批已签名授权，使用时直接取用
2. **CAW 硬件加速**：Cobo 企业级部署可缩短 MPC 协调时间
3. **批量支付**：将多次小额查询合并为一次支付请求

对于 Agent-Native Payments 场景，用户理解 AI 在"思考和操作"，15 秒是合理的心理预期范围。

---

### Q5. 会话预算上限是如何实现的？超限后会怎样？

**A：** 在 `run_agent()` 函数中，每次工具调用前检查"已支出 + 本次费用"是否超过 `max_spend` 上限（默认 0.05 USDC / 会话）。

超限时，Agent 不会执行工具调用，而是向 LLM 返回一条错误消息："会话预算已达上限，本次查询取消"。LLM 会据此生成一个友好的回复告知用户，而不是静默失败或抛出异常。

这个机制有两层意义：
- **用户保护**：防止单次对话因 LLM 误判而触发大量查询
- **演示可控**：评审时可以清晰展示"Agent 有边界意识"

---

### Q6. 如何保证支付安全？有没有风险控制？

**A：** 安全防护分两个层次，但需要坦诚说明各自的边界。

**Pact 策略层（MPC 节点执行，不可绕过）：**

当前 Pact 做了类型级约束——只允许对 Base Sepolia USDC 合约的 `TransferWithAuthorization` EIP-712 消息签名。攻击者即使控制了 Agent 代码，也无法用这个 Pact key 去签名转 ETH、调用其他合约、或在主网操作。

**但需要坦诚的局限：** 当前 Pact 没有配置频率限制（`deny_if.usage_limits.request_count_gt`）。如果 Agent 代码被攻破，攻击者可以循环发起合法类型的签名请求，每次指定大额 `value` 转给自己——类型合规，Pact 会放行。

**Agent 代码层（`SESSION_BUDGET_USDC = 0.05`）：**

每会话限制 $0.05 USDC 支出，超限后 Agent 主动拒绝工具调用。这提供了日常使用的保护，但一旦 Agent 代码被控制，这层检查也同时失效。

**生产环境的正确做法：** 在 Pact 中加入 `deny_if.usage_limits`（如每小时最多 20 次签名请求），让频率限制在 MPC 节点层执行，使其不依赖 Agent 代码的完整性。当前 Demo 作为测试网演示未配置此项，使用真实资金时必须补上这一层。

**如何证明支付真实发生：** 每笔支付完成后终端显示完整 tx hash，粘贴到 `sepolia.basescan.org/tx/` 可查验真实 USDC 转账记录；会话摘要的余额差值精确等于成功支付次数 × $0.01。

---

### Q7. 支付失败怎么处理？有幂等性保证吗？

**A：** 支付层面有两道保护：

**x402 层**：每笔支付使用随机生成的 `nonce`（bytes32），同一 nonce 的授权只能被 Facilitator 使用一次，天然防止重放攻击。

**Agent 层**：工具调用失败后有一次静默重试（等待 1.5s），重试使用全新的 nonce 和签名，不会影响已失败的请求。UI 只展示最终结果，隐藏中间状态。

偶发的 Facilitator 拒绝（返回 402）不会导致资金损失，因为链上转账只在 Facilitator 验证通过后才执行。

---

### Q8. 这个方案和"给 Agent 充值一个 API Key"有什么本质区别？

**A：** 核心区别在于**支付的原子性和可验证性**。

API Key 方案：资金预先托管在平台，调用是否真实发生、费用是否公平，完全依赖平台的记账系统，用户无法独立验证。

x402 + 链上支付方案：
- **按需支付**：每次查询触发一笔独立的链上交易，没有预充值
- **链上可验证**：任何人都可以查看资金流向，无法造假
- **无需信任第三方**：Facilitator 只是执行者，资金直接从 Agent 钱包到服务方钱包
- **天然审计**：链上记录就是完整的支付审计日志

这正是 Web3 支付相比 Web2 的根本优势——**Trust, but verify on-chain**。

---

### Q9. 这个架构在主网上能用吗？需要哪些改动？

**A：** 架构本身已经是生产级设计，上主网主要需要三处调整：

1. **网络切换**：将 `eip155:84532`（Base Sepolia）改为 `eip155:8453`（Base 主网），USDC 合约地址相应替换
2. **CAW 配置**：使用主网 Pact，verifyingContract 指向主网 USDC
3. **Facilitator**：x402.org 目前仅支持 Base Sepolia，主网需使用 Coinbase 的生产 Facilitator 或自建

代码层面无需大改，`NETWORK` 常量和合约地址是唯一的硬编码配置。

---

### Q10. 如果 Cobo Agentic Wallet 服务不可用，Agent 会怎样？

**A：** Agent 会优雅降级而非崩溃。

当 CAW MPC 签名超时（120s）或返回失败状态时，`CawSigner` 抛出异常，`_execute_tool` 捕获后返回带错误信息的 `PaymentEvent`。Agent 静默重试一次，若仍失败则向 LLM 返回工具错误，LLM 会告知用户"当前无法获取该数据"。

整个系统的故障边界清晰：CAW 不可用 → 工具调用失败 → LLM 感知并回复 → 用户得到明确反馈。不会出现无限等待、余额被错误扣除或程序崩溃的情况。
